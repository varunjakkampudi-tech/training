Overview

Cross-browser compatibility means your site works consistently across browsers, versions, devices, and operating systems — despite differences in rendering engines and feature support.

  • Goal: a consistent, functional experience — not pixel-identical everywhere.
  • Variables: engine, version, OS, device, viewport, network, settings.
  • Strategy: standards + feature detection + graceful fallbacks + testing.
Interview importance: Feature detection vs browser sniffing, polyfills vs transpiling, vendor prefixes, and responsive strategies are the core topics.

Why Compatibility Bugs Happen

  • Different engines implement specs at different times / interpret edge cases differently.
  • New features ship first behind flags/prefixes; older browsers lack them.
  • Default styles (user-agent stylesheets) differ.
  • Bugs & legacy quirks (old IE, Safari date parsing, flexbox gaps).
  • Device differences — viewport, DPI, touch, fonts.

Feature Detection (not Browser Sniffing)

✅ Feature detection

test the capability
if ("IntersectionObserver" in window) { /* use it */ }
if (CSS.supports("display", "grid")) { /* grid */ }

❌ Browser sniffing

fragile
if (navigator.userAgent.includes("Chrome")) { ... }
// breaks constantly; UA strings lie
CSS feature queries
@supports (display: grid) {
  .layout { display: grid; }
}
@supports not (gap: 1rem) {
  .layout > * { margin: 0.5rem; } /* fallback */
}
Rule: Detect features, not browsers. Use @supports in CSS and "x" in window / CSS.supports() in JS.

Polyfills & Transpiling

Polyfill

Adds a missing API at runtime (e.g. fetch, Promise, IntersectionObserver).

Transpile

Rewrites new syntax to older syntax at build time (Babel: optional chaining → ES5).

Conditional (only load if needed)
if (!("fetch" in window)) {
  await import("whatwg-fetch"); // polyfill on demand
}

Vendor Prefixes

Prefixes (usually auto-generated)
.box {
  -webkit-user-select: none; /* Safari */
  user-select: none;
  -webkit-backdrop-filter: blur(4px); /* Safari */
  backdrop-filter: blur(4px);
}
Don't hand-write prefixes. Use Autoprefixer (PostCSS) driven by Browserslist — it adds only what your targets need.

Progressive Enhancement & Graceful Degradation

Progressive Enhancement

Start with a baseline that works everywhere, then layer on enhancements where supported.

Graceful Degradation

Build for modern browsers, then ensure it still works (reduced) on older ones.

Prefer progressive enhancement: core content/functionality in HTML first; CSS & JS enhance. If JS fails, the page still works.

Reset / Normalize CSS

Browsers ship different default (user-agent) styles. Normalize/reset makes the baseline consistent.

Modern baseline
*, *::before, *::after { box-sizing: border-box; }
* { margin: 0; }
img, picture, video { max-width: 100%; display: block; }
input, button { font: inherit; }
Normalize.css preserves useful defaults and fixes bugs; a reset strips everything. Modern projects often use a small custom reset.

Responsive Design

Mobile-first + fluid
<meta name="viewport" content="width=device-width, initial-scale=1">

.grid { grid-template-columns: 1fr; }
@media (min-width: 768px) { .grid { grid-template-columns: repeat(3, 1fr); } }

/* fluid without breakpoints */
.title { font-size: clamp(1.5rem, 4vw, 3rem); }
  • Mobile-first — min-width media queries; start small.
  • Fluid type/space with clamp(); responsive images with srcset/sizes.
  • Container queries for component-level responsiveness (check support).
  • Touch targets, hover-not-guaranteed (@media (hover: hover)).

JavaScript Compatibility

  • Transpile modern syntax (Babel + preset-env by Browserslist).
  • Polyfill missing APIs (core-js, fetch, IntersectionObserver, ResizeObserver).
  • Feature-detect before use; provide fallbacks.
  • Ship modern + legacy bundles (<script type="module"> + nomodule).
module/nomodule pattern
<script type="module" src="modern.js"></script>
<script nomodule src="legacy.js"></script>

Best Practices

  • Write standards-compliant, semantic HTML/CSS.
  • Feature-detect, never browser-sniff.
  • Progressive enhancement; graceful fallbacks (@supports).
  • Autoprefixer + Babel driven by Browserslist.
  • Normalize baseline; mobile-first responsive.
  • Test on real WebKit/Safari, not just Chrome.
  • Support browsers your analytics show; don't over-support.

Scenario-Based Questions

Scenario 1 — Works in Chrome, broken in Safari CSS/JS

Q: A feature works in Chrome but not iOS Safari.

Show approach

Test in real WebKit. Check caniuse for the CSS/JS feature; add prefixes (Autoprefixer), polyfill missing APIs, and use @supports/feature detection with a fallback. Watch 100vh, date parsing, and backdrop-filter.

Scenario 2 — Layout breaks on old browsers CSS

Q: CSS Grid layout collapses on an older browser.

Show approach

Wrap grid in @supports (display:grid) with a fla fallback (flex/float) in the base rule. Progressive enhancement ensures a usable layout everywhere.

Scenario 3 — JS error only on one browser JS

Q: A newer JS method throws on an older browser.

Show approach

Transpile with Babel/preset-env and polyfill the missing built-in (core-js). Verify Browserslist targets. Feature-detect before using cutting-edge APIs.

Top Interview Questions

What is cross-browser compatibility?

Ensuring a site works consistently across browsers, versions, and devices despite differences in engines and feature support.

Feature detection vs browser sniffing?

Feature detection tests whether a capability exists ("x" in window, CSS.supports, @supports) — robust. Browser sniffing parses the UA string — fragile and unreliable.

Polyfill vs transpiling?

A polyfill adds a missing runtime API; transpiling (Babel) rewrites new syntax to older syntax at build time.

What are vendor prefixes and how do you manage them?

Engine-specific property prefixes (-webkit-, -moz-). Manage automatically with Autoprefixer driven by Browserslist.

Which engines do major browsers use?

Blink (Chrome/Edge/Opera), WebKit (Safari + all iOS browsers), Gecko (Firefox).

Why must you test Safari specifically?

WebKit lags/differs on some features, and all iOS browsers use WebKit — so Chrome desktop passing doesn't guarantee iOS works.

Progressive enhancement vs graceful degradation?

PE builds a working baseline then layers enhancements; GD builds for modern then ensures it still degrades acceptably. PE is generally preferred.

How do you handle the mobile 100vh issue?

Mobile browsers change viewport height with the address bar; use dvh/svh/lvh units (with a vh fallback).

What is Browserslist?

A shared config declaring target browsers; Babel, Autoprefixer, and other tools read it to decide what to transpile/prefix/polyfill.

Normalize vs reset CSS?

Reset strips all default styles; Normalize keeps useful defaults and fixes cross-browser bugs. Both create a consistent baseline.

How do you deliver modern + legacy JS?

<script type="module"> loads a modern bundle; nomodule loads a transpiled legacy bundle for older browsers.

What is @supports?

A CSS feature query that applies rules only if the browser supports a property/value — enabling fallbacks without JS.

How do you decide which browsers to support?

Analytics-driven: support what your users actually use, plus baseline standards; declare it in Browserslist. Avoid over-supporting dead browsers.

How do you test across browsers efficiently?

Playwright (Chromium/Firefox/WebKit) for automation, cloud device labs (BrowserStack) for real devices, plus manual spot checks on top browsers.

How do you handle unsupported CSS features?

Provide a base fallback then enhance inside @supports; use Autoprefixer for prefixes; test with caniuse data.
