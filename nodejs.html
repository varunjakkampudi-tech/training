Overview

Node.js is a JavaScript runtime built on Chrome's V8 engine. It's single-threaded, event-driven, and non-blocking — ideal for I/O-heavy, real-time applications.

  • Non-blocking I/O — handles thousands of concurrent connections with one thread.
  • Event-driven — callbacks/promises fire when I/O completes.
  • Single language — JS on both client and server.
  • Huge ecosystem — npm.
Not ideal for: CPU-bound work (image/video processing, heavy computation) — it blocks the event loop. Offload to worker threads or separate services.

Event Loop & libuv Phases

PhaseHandles

pending callbacksdeferred I/O callbacks
pollretrieve new I/O events; execute I/O callbacks

nextTick & microtasks run between phases
process.nextTick(() => console.log("nextTick")); // highest priority
Promise.resolve().then(() => console.log("promise"));
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// nextTick -> promise -> timeout/immediate (order varies by context)
Bottleneck: A synchronous CPU-heavy loop blocks the entire event loop — all requests stall. Never do heavy sync work in a request handler.

Async Patterns

Callbacks → Promises → async/await
const fs = require("fs/promises");

async function read() {
  try { return await fs.readFile("a.txt", "utf8"); }
  catch (e) { console.error(e); }
}

// Promisify a callback API
const { promisify } = require("util");
const sleep = promisify(setTimeout);

EventEmitter

The pub/sub backbone of Node — streams, HTTP servers, and sockets are all emitters.

EventEmitter
const EventEmitter = require("events");
const bus = new EventEmitter();
bus.on("order", (id) => console.log("got", id));
bus.emit("order", 42);
Watch out: "MaxListenersExceededWarning" signals a leak — you're adding listeners without removing them.

Streams & Buffers

Streams process data in chunks instead of loading it all in memory — essential for large files and network data.

TypeExample
Readablefs.createReadStream, HTTP request
Writablefs.createWriteStream, HTTP response
DuplexTCP socket
Transformzlib gzip, crypto
Pipe with backpressure
const { pipeline } = require("stream");
pipeline(
  fs.createReadStream("big.log"),
  zlib.createGzip(),
  fs.createWriteStream("big.log.gz"),
  (err) => err && console.error(err)
);
Backpressure: pipe/pipeline automatically pause the source when the destination is slow — prevents memory blowups.

Cluster (Multi-Core)

Node is single-threaded, so a process uses one CPU core. The cluster module forks workers to use all cores, sharing the same port.

cluster
const cluster = require("cluster");
const os = require("os");
if (cluster.isPrimary) {
  os.cpus().forEach(() => cluster.fork());
} else {
  require("./server"); // each worker runs the app
}
In production use a process manager like PM2 (cluster mode, restarts, logs) instead of hand-rolling clustering.

Caching & Redis

Cache-aside with Redis
async function getUser(id) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);
  const user = await db.findUser(id);
  await redis.set(`user:${id}`, JSON.stringify(user), "EX", 300);
  return user;
}
Use Redis for: caching, sessions, rate limiting, pub/sub, queues, distributed locks.

Error Handling

Central handler + global safety nets
// Express error middleware (4 args)
app.use((err, req, res, next) => {
  logger.error(err);
  res.status(err.status || 500).json({ error: err.message });
});

process.on("unhandledRejection", (e) => logger.error(e));
process.on("uncaughtException", (e) => { logger.error(e); process.exit(1); });
After an uncaughtException the process is in an undefined state — log and restart (let PM2/Docker relaunch it). Don't just resume.

Logging

  • Use structured JSON logs (pino, winston) with levels (info/warn/error).
  • Add a request ID (correlation ID) for tracing across services.
  • Never log secrets/PII; ship logs to a central store (ELK, CloudWatch).

File Upload

Multer
const upload = multer({ limits: { fileSize: 5 * 1024 * 1024 } });

Scenario-Based Questions

Scenario 1 — API is slow under load Performance

Q: Response times spike as traffic grows. How do you diagnose & fix?

Show approach

Profile (clinic.js / --prof). Check for blocking sync code, missing DB indexes, and N+1 queries. Add caching (Redis), connection pooling, clustering/PM2 across cores, and a load balancer. Move CPU work to worker threads.

Scenario 2 — Memory keeps growing Memory

Q: RSS climbs until the process crashes. How do you find the leak?

Show approach

Take heap snapshots (--inspect), compare growth. Common causes: unbounded caches/arrays, event listeners not removed, closures holding references, global state. Fix with limits, cleanup, and WeakMap.

Scenario 3 — CPU-bound endpoint blocks everything Event Loop

Q: One report-generation endpoint freezes the whole server.

Show approach

The sync work blocks the single-threaded event loop. Offload to a worker thread or a separate queue/worker service (Bull + Redis), and stream results.

Production Issues

Unhandled rejection crashes the app Bug

Report: Process exits randomly.

Show fix

Add .catch/try-catch to every async path, a global unhandledRejection handler for logging, and run under a supervisor (PM2/Docker) for auto-restart.

"EADDRINUSE" on deploy Bug

Report: Server won't start — port in use.

Show fix

A previous process didn't exit. Implement graceful shutdown on SIGTERM (close server + DB), and ensure the orchestrator waits for exit before restart.

Top Interview Questions

Is Node.js single-threaded?

Your JS runs on a single thread with an event loop, but libuv uses a background thread pool for some async I/O (fs, crypto, DNS). CPU work should go to worker threads.

Explain the event loop phases.

timers → pending callbacks → poll → check (setImmediate) → close. process.nextTick and promise microtasks run between phases, nextTick first.

setImmediate vs setTimeout vs process.nextTick?

nextTick runs before other microtasks; setImmediate runs in the check phase; setTimeout(0) in the timers phase. nextTick can starve I/O if overused.

What are streams and why use them?

Interfaces to process data in chunks (readable/writable/duplex/transform), enabling low memory for large files/network data with automatic backpressure.

Cluster vs worker threads?

Cluster forks separate processes to scale I/O across cores (shared port). Worker threads share memory and are for CPU-bound tasks within a process.

How do you handle errors in async code?

try/catch with async/await, .catch on promises, Express error middleware, and global unhandledRejection/uncaughtException handlers (log + restart).

What is middleware?

Functions (req, res, next) that run in order to parse, authenticate, log, or handle errors before/around route handlers.

How do you secure a Node API?

helmet headers, CORS allow-list, input validation, parameterized queries, JWT/bcrypt auth, rate limiting, HTTPS, secrets in env, npm audit.

package.json vs package-lock.json?

package.json declares dependencies with ranges; the lockfile pins the exact resolved tree for reproducible installs (npm ci).

How do you scale a Node app?

Cluster/PM2 across cores, horizontal scaling behind a load balancer, caching (Redis)/CDN, stateless services, connection pooling, and offloading CPU/async jobs to queues.

What is backpressure?

When a writable stream can't keep up with a readable source; pipe/pipeline pause the source to avoid unbounded memory growth.

Callback hell — how to avoid?

Use promises and async/await, modularize functions, and Promise.all for parallelism.

What is the difference between exports and module.exports?

module.exports is the actual exported object; exports is a reference to it. Reassigning exports breaks the link — assign to module.exports.

How do you do graceful shutdown?

Listen for SIGTERM/SIGINT, stop accepting new connections (server.close), drain in-flight requests, close DB/Redis, then exit.

What is the thread pool used for?

libuv's pool (default 4, UV_THREADPOOL_SIZE) runs blocking ops like file I/O, DNS, crypto, and compression off the main thread.
