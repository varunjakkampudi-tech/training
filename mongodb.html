1 Introduction

MongoDB is an open-source, document-oriented NoSQL database that stores data as flexible, JSON-like documents.

Instead of tables, rows, and columns, MongoDB organizes data into collections of documents. Each document is a set of key–value pairs stored physically as BSON (Binary JSON), which extends JSON with extra types like Date, ObjectId, and Binary.

SQL TermMongoDB Term
DatabaseDatabase
TableCollection
Row / RecordDocument
ColumnField
JOIN$lookup / embedding
Primary Key_id (ObjectId)
Note: BSON = Binary JSON — a binary-encoded serialization of JSON-like documents that enables fast traversal and rich data types not available in plain JSON.

2 SQL (MySQL) vs MongoDB

Why teams move from relational databases to MongoDB — and the trade-offs involved.

⚙️ MySQL (RDBMS)

  • Relational — data split across normalized tables
  • Fixed schema — structure defined up front, slower to change
  • Vertical scaling — harder to scale out
  • JOINs are powerful but CPU-heavy at scale
  • Strong ACID guarantees
  • Uses SQL (Structured Query Language)

🍃 MongoDB (NoSQL)

  • Document-oriented — related data can live together
  • Dynamic schema — flexible, fast to evolve
  • Horizontal scaling via sharding
  • Avoids costly JOINs by embedding data
  • ACID at document level; multi-doc transactions since 4.0
  • Rich query API + aggregation framework
ACID = Atomicity, Consistency, Isolation, Durability. MongoDB provides these at the single-document level by default, and across multiple documents using transactions.

🌍 Real-world scenario

An e-commerce catalog where products have wildly different attributes (a laptop has RAM/CPU, a shirt has size/color) fits MongoDB's flexible documents better than dozens of sparse SQL columns or complex join tables.

Insert one
db.products.insertOne({
  name: "Laptop",
  price: 899.99,
  stock: 10,
  tags: ["sale", "electronics"]
})
Insert many

7 Read (Find)

Basic reads
db.products.find()                 // all documents
db.products.findOne({ name: "Laptop" })  // first match
db.products.find().pretty()        // formatted output

Sorting, limiting, skipping, counting

Cursor methods
db.products.find().sort({ stock: -1 })   // 1 = asc, -1 = desc
db.products.find().limit(2)           // only 2 docs
db.products.find().skip(2)            // skip first 2 (pagination)
db.products.countDocuments()          // count results

Projection — choosing fields

Projection
// 1 = include, 0 = exclude
db.products.find({}, { name: 1, price: 1, _id: 0 })
Pagination pattern: for large collections prefer range-based paging (_id > lastSeenId) — skip gets slow on big offsets.

8 Update

Update basics
db.products.updateOne(
  { _id: 1 },                    // filter
  { $set: { name: "Product 4" } }   // update
)

db.products.updateMany(
  { stock: { $lt: 5 } },
  { $set: { status: "low" } }
)
Upsert: Add { upsert: true } as a third argument to insert if no match exists, otherwise update.

Field update operators

OperatorPurpose
$setSet a field's value
$incIncrement / decrement a number
$renameRename a field
$unsetRemove a field
$currentDateSet field to current date
$mulMultiply a field's value
$min / $maxUpdate only if smaller / larger
Field operators
db.products.updateOne({ _id: 1 }, { $inc: { stock: -2 } })       // sell 2 units
db.products.updateOne({ _id: 1 }, { $rename: { since: "liveSince" } })
db.products.updateOne({ _id: 1 }, { $unset: { details: "" } })  // value ignored
db.products.updateOne({ _id: 1 }, { $currentDate: { lastModified: true } })

Array update operators

OperatorPurpose
$pushAppend an element
$addToSetAppend only if not already present
$popRemove first (-1) or last (1)
$pullRemove all elements matching a condition
$eachPush / add multiple elements at once
Array operators
db.products.updateOne({ _id: 1 }, { $push: { ratings: 9 } })
db.products.updateOne({ _id: 1 }, { $addToSet: { tags: "durable" } })
db.products.updateOne({ _id: 1 }, { $pop: { ratings: 1 } })   // remove last
db.products.updateOne({ _id: 1 }, { $pull: { ratings: 9 } })  // remove all 9s

🌍 Real-world scenario

Inventory: $inc: { stock: -1 } per purchase. Likes: $addToSet stops a user liking twice. Activity feed: $push + $slice keeps only the latest N events.

9 Delete

Delete
db.products.deleteOne({ name: "John" })       // first match
db.products.deleteMany({ stock: { $eq: 0 } })  // all matches
db.products.deleteMany({})                     // ⚠️ deletes everything
Danger: deleteMany({}) with an empty filter removes every document. Always double-check the filter.

12 Querying Arrays

Array matching
// Match if "sale" is ONE OF the elements
db.products.find({ tags: "sale" })

// Match ONLY the exact array [ "sale" ]
db.products.find({ tags: ["sale"] })

// Must contain BOTH values ($all)
db.products.find({ tags: { $all: ["sale", "discount"] } })

// Array size + combined element conditions
db.products.find({ tags: { $size: 3 } })
db.orders.find({ items: { $elemMatch: { qty: { $gt: 1 }, price: { $lt: 50 } } } })
$elemMatch matters when a single array element must satisfy multiple conditions at once — not just any elements individually.

13 Aggregation Pipeline

A pipeline transforms documents through an ordered series of stages; each stage's output feeds the next — like an assembly line.

Pipeline example
db.posts.aggregate([
  // Stage 1: keep only popular posts
  { $match: { likes: { $gt: 1 } } },
  // Stage 2: group by category, summing likes
  { $group: { _id: "$category", totalLikes: { $sum: "$likes" } } },
  // Stage 3: highest totals first
  { $sort: { totalLikes: -1 } }
])

Common stages

StageWhat it does
$matchFilter documents (like find)
$groupGroup & aggregate ($sum, $avg, $max…)
$sortOrder results
$projectReshape / pick fields
$addFieldsAdd computed fields
$limit / $skipPaginate
$countCount documents
$unwindFlatten an array into multiple docs
$lookupLeft outer join with another collection
$out / $mergeWrite results to a collection
$group + $sort
// Average price per category, richest first
db.products.aggregate([
  { $group: { _id: "$category", avgPrice: { $avg: "$price" }, n: { $sum: 1 } } },
  { $sort: { avgPrice: -1 } }
])

$lookup — joining collections

$lookup
db.comments.aggregate([
  { $lookup: {
      from:         "movies",        // collection to join
      localField:   "movie_id",     // field in comments
      foreignField: "_id",          // field in movies
      as:           "movie_details"  // output array field
  }},
  { $limit: 1 }
])

🌍 Real-world scenario

Sales dashboard: $match this month's orders → $unwind line items → $group revenue by product → $sort top sellers → $lookup product names → $merge into a reports collection for BI.

14 Indexes

Indexes make queries fast by avoiding full collection scans — the single biggest performance lever in MongoDB.

Index management
db.products.createIndex({ name: 1 })                // single field
db.products.createIndex({ category: 1, price: -1 }) // compound
db.products.createIndex({ email: 1 }, { unique: true })
db.products.createIndex({ description: "text" })     // full-text

db.products.getIndexes()
db.products.find({ name: "Laptop" }).explain("executionStats")

Index types

TypeDefinitionBest for
Single field{ name: 1 }Equality / sort on one field
Compound{ category: 1, price: -1 }Multi-field filters + sort
Multikey{ tags: 1 }Indexing array fields
Text{ body: "text" }Full-text search
Hashed{ userId: "hashed" }Even sharding distribution
TTL{ createdAt: 1 }, { expireAfterSeconds }Auto-expiring data (sessions, logs)
Partial{ partialFilterExpression }Index only a subset → smaller/cheaper
Unique{ email: 1 }, { unique: true }Enforce no duplicates

Covered queries (fastest possible)

Covered query
// Index includes every field the query needs (filter + projection)
db.users.createIndex({ email: 1, name: 1 })

// Served ENTIRELY from the index — never touches the documents
db.users.find({ email: "a@x.com" }, { name: 1, _id: 0 })
ESR rule for compound indexes: order fields as Equality → Sort → Range.
Trade-off: Indexes speed reads but slow writes and use RAM/storage. Index what you actually query — every extra index is paid for on every insert/update.

15 Performance Bottlenecks & Cost Optimization

How to diagnose slow queries, remove the common bottlenecks, and cut your database bill — the levers that matter in production.

Diagnose first — explain()

Read the query plan
db.orders.find({ status: "paid" }).explain("executionStats")

// Look at:
//   stage: COLLSCAN  -> BAD (full scan, add an index)
//   stage: IXSCAN    -> GOOD (using an index)
//   totalDocsExamined vs nReturned -> want these close

Common bottlenecks & fixes

BottleneckSymptomFix
Missing indexCOLLSCAN, high CPUAdd a matching index (ESR order)
Unbounded array growthDocs approach 16 MB, slow readsBucket data or reference instead of embed
Large skip() pagingDeep pages get slowerRange-based paging (_id > lastId)
N+1 queriesMany round-trips per request$lookup or embed related data
Returning whole docsHigh network / RAM useProjection — fetch only needed fields
Working set > RAMConstant disk reads, page faultsAdd RAM, index smarter, or shard
Poor shard keyOne "hot" shard, uneven loadHigh-cardinality / hashed shard key
Over-indexingSlow writes, high storage costDrop unused indexes ($indexStats)
Find unused indexes to drop
db.orders.aggregate([{ $indexStats: {} }])
// accesses.ops === 0  -> index is unused -> drop it to save writes & storage
db.orders.dropIndex("staleIndex_1")

🌍 Real use cases — better performance

  • Login by email: a unique index on email turns a full scan into an instant index lookup.
  • Product listing page: compound index { category: 1, price: -1 } serves "filter by category, sort by price" in one indexed pass.
  • Dashboard counts: pre-aggregate with $merge into a summary collection nightly instead of scanning millions of rows per page load.
  • Autocomplete: a text (or Atlas Search) index answers prefix queries in milliseconds.

17 Data Modeling & Schema Design

The most important skill in MongoDB. Good models make queries fast and simple; bad ones cause slow queries, huge documents, and pain. "Data that is accessed together should be stored together."

Embedding vs Referencing

📎 Embedding (nesting)

  • Store related data inside the parent document
  • Pro: one read fetches everything — fastest
  • Pro: atomic single-document updates
  • Con: can duplicate data; risks the 16 MB limit
  • Best for one-to-one & one-to-few "contains"

🔗 Referencing (linking)

  • Store an _id pointing to another document
  • Pro: no duplication; small documents
  • Pro: unbounded growth is safe
  • Con: needs extra queries / $lookup
  • Best for one-to-many & many-to-many
Decision rule: Embed if the data is read together, bounded in size, and mostly updated together. Reference if it grows without limit, is large, or is shared across many parents.

Relationship types

One-to-One — embed

User + profile embedded
{
  _id: ObjectId, name: "Aria",
  profile: { bio: "Dev", avatar: "a.png" }  // 1:1 -> embed
}

One-to-Many — embed if "few", reference if "many"

Post with few comments (embed) vs many (reference)
// One-to-FEW: embed
{ _id: ObjectId, title: "Hello", comments: [ { user, text } ] }

// One-to-MANY (unbounded): reference from the "many" side
// comments collection
{ _id: ObjectId, postId: ObjectId, user: "...", text: "..." }

Many-to-Many — array of references

Students ↔ Courses
// student holds course ids (and/or vice-versa)
{ _id: ObjectId, name: "Sam", courseIds: [ ObjectId, ObjectId ] }

Schema design patterns (senior interview favorites)

PatternProblem it solvesHow
BucketUnbounded time-series/IoT arraysGroup many readings into one doc per hour/day
AttributeMany rare/variable fields to indexStore as array of { k, v } pairs, index k&v
SubsetHuge related lists but only need a fewEmbed the top-N; keep the rest referenced
OutlierA few docs are far larger than the restFlag outliers; overflow to a side collection
Extended ReferenceRepeated $lookup for a few fieldsCopy the few needed fields into the parent
ComputedExpensive repeated calculationsPre-compute & store (e.g. likeCount)
TreeHierarchies (categories, org charts)Parent refs, child refs, or materialized paths
Bucket pattern — sensor data
{
  sensorId: "s-1",
  hour: ISODate("2026-07-21T10:00:00Z"),
  count: 60,
  readings: [ { t: "10:00", v: 21.3 }, { t: "10:01", v: 21.4 } ]
}  // 60 readings in ONE doc instead of 60 docs
Attribute pattern — variable product specs
{
  name: "Laptop",
  specs: [ { k: "ram", v: "16GB" }, { k: "cpu", v: "i7" } ]
}  // one index on specs.k + specs.v covers ALL attributes
Tree pattern — categories (materialized path)
{ _id: "electronics",        path: "," }
{ _id: "laptops", parent: "electronics", path: ",electronics," }
// find all descendants of electronics:
db.categories.find({ path: { $regex: /,electronics,/ } })

🌍 Real-world scenario

E-commerce order: embed the line items (read together, bounded) but use an extended reference to copy just productName + price into each item — so rendering an order never needs a $lookup, and product edits don't rewrite history.

Anti-pattern: Unbounded arrays (e.g. embedding every comment forever) eventually hit the 16 MB limit and slow every read. Use referencing or the bucket/subset pattern instead.

18 Transactions

Multi-document ACID transactions (v4.0+, sharded in v4.2+) let several operations across documents/collections succeed or fail together. Require a replica set.

The four building blocks

StepPurpose
startSession()Create a session that groups the operations
startTransaction()Begin the transaction on that session
commitTransaction()Persist all changes atomically
abortTransaction()Roll everything back on any error

Real banking example — transfer money

Atomic transfer with rollback
const session = client.startSession();
try {
  session.startTransaction();
  const accounts = client.db("bank").collection("accounts");

  // 1) Debit A only if it has enough balance (guard!)
  const debit = await accounts.updateOne(
    { _id: "A", balance: { $gte: 100 } },
    { $inc: { balance: -100 } },
    { session }
  );
  if (debit.modifiedCount === 0) throw new Error("Insufficient funds");

  // 2) Credit B
  await accounts.updateOne({ _id: "B" }, { $inc: { balance: 100 } }, { session });

  await session.commitTransaction();   // both succeed together
} catch (e) {
  await session.abortTransaction();    // neither happens
  throw e;
} finally {
  await session.endSession();
}
Best practice: Keep transactions short, touch as few documents as possible, and prefer a single atomic update where you can — single-document writes are already atomic and far cheaper than a transaction.
Bottleneck: Long-running transactions hold locks and can abort under write conflicts. Don't do slow work (HTTP calls, heavy loops) inside a transaction.

19 Replication (High Availability)

A replica set is a group of mongod nodes holding the same data, giving you redundancy and automatic failover.

RoleResponsibility
PrimaryReceives all writes; the single source of truth
SecondaryReplicates the primary's oplog; can serve reads
ArbiterVotes in elections but holds no data (tie-breaker)

Election

If the primary becomes unreachable, the remaining members hold an election and promote a secondary to primary — usually within a few seconds. Drivers detect the new primary and reconnect automatically, so the app keeps running.

Read preference & write concern
// Read from secondaries to offload analytics
db.orders.find().readPref("secondaryPreferred")

// Wait until a majority of nodes acknowledge the write (durable)
db.orders.insertOne(doc, { writeConcern: { w: "majority", j: true } })
SettingOptionsMeaning
Write concernw:1, w:"majority", j:trueHow many nodes / journal must ack a write
Read preferenceprimary, secondaryPreferred, nearestWhich node serves reads
Interview point: w:"majority" + read concern "majority" gives you strong consistency; reading from secondaries with w:1 can return slightly stale data (eventual consistency).

20 Sharding (Horizontal Scaling)

Sharding splits one large collection across multiple servers so you can scale beyond a single machine's storage and write throughput.

ComponentRole
ShardA replica set holding a subset of the data
Shard keyField(s) that decide which shard a doc lives on
ChunkA contiguous range of shard-key values (default 128 MB)
BalancerMoves chunks between shards to keep them even
mongosQuery router the app connects to
Config serversStore cluster metadata (chunk → shard map)
Enable sharding
sh.enableSharding("shop")
// hashed key -> even write distribution
sh.shardCollection("shop.orders", { customerId: "hashed" })

Choosing a shard key

  • High cardinality — many distinct values so data can split finely.
  • Low frequency — no single value dominates (avoids a "hot" chunk).
  • Non-monotonic — avoid ever-increasing keys (like timestamps) that send all new writes to one shard. Use hashed keys instead.
Bottleneck — shard imbalance / hot partition: A poor shard key concentrates traffic on one shard while others idle. Fix with a hashed or compound key that spreads writes evenly.

21 CAP Theorem

A distributed system can guarantee only two of three: Consistency, Availability, Partition tolerance. Since network partitions are unavoidable, real systems choose between C and A during a partition.

LetterMeaning
C — ConsistencyEvery read sees the most recent write
A — AvailabilityEvery request gets a (non-error) response
P — Partition toleranceThe system keeps working despite network splits
Where MongoDB sits: MongoDB is CP by default — during a partition it favors consistency (only the primary accepts writes; a minority side steps down). You can tune toward availability with read preferences and write concerns, trading off freshness.

22 BSON Internals & ObjectId

BSON (Binary JSON) is how MongoDB stores documents on disk and on the wire.

Why BSON over JSON?

  • Faster — length-prefixed fields allow skipping without parsing every byte.
  • Typed — supports Date, ObjectId, Decimal128, Binary, int/long that JSON lacks.
  • Traversable — designed for efficient indexing and partial reads.

ObjectId structure (12 bytes)

BytesPartPurpose
4TimestampCreation time (seconds since epoch) — roughly sortable
5Random valueUnique per machine/process
3CounterIncrementing, seeded randomly — avoids collisions
Use the embedded timestamp
const id = new ObjectId();
id.getTimestamp();   // Date the document was created
// Sorting by _id ≈ sorting by creation time
db.events.find().sort({ _id: -1 }).limit(10)  // newest 10

23 Advanced Aggregation

Beyond $match/$group — the power stages that show up in analytics and senior interviews.

StageWhat it does
$facetRun several sub-pipelines in parallel (multi-metric dashboards)
$bucketGroup docs into user-defined ranges (histograms)
$bucketAutoAuto-compute evenly distributed buckets
$graphLookupRecursive lookups (org charts, social graphs)
$setWindowFieldsWindow functions — running totals, moving averages, rank
$replaceRoot / $replaceWithPromote a sub-document to the top level
$unionWithCombine results from another collection
$densifyFill gaps in a numeric/date sequence
$fillFill missing field values (carry-forward/linear)
$facet — dashboard in one query
db.orders.aggregate([{
  $facet: {
    totalRevenue: [ { $group: { _id: null, v: { $sum: "$amount" } } } ],
    topProducts:  [ { $sortByCount: "$product" }, { $limit: 5 } ],
    byStatus:     [ { $group: { _id: "$status", n: { $sum: 1 } } } ]
  }
}])
$setWindowFields — running total
db.sales.aggregate([{
  $setWindowFields: {
    partitionBy: "$region",
    sortBy: { date: 1 },
    output: { runningTotal: { $sum: "$amount",
      window: { documents: [ "unbounded", "current" ] } } }
  }
}])
$graphLookup — employee → all managers
db.employees.aggregate([{
  $graphLookup: {
    from: "employees",
    startWith: "$managerId",
    connectFromField: "managerId",
    connectToField: "_id",
    as: "managementChain"
  }
}])
Memory: Each stage has a 100 MB limit. For big sorts/groups add { allowDiskUse: true }.

24 explain() & Query Plans

The tool that tells you exactly how MongoDB runs a query — essential for tuning.

Three verbosity modes
db.orders.find({ status: "paid" }).explain("queryPlanner")      // chosen plan
db.orders.find({ status: "paid" }).explain("executionStats")   // + real timing
db.orders.find({ status: "paid" }).explain("allPlansExecution") // all candidates

Reading the plan — key stages

StageMeaningVerdict
COLLSCANFull collection scan❌ Bad — add an index
IXSCANIndex scan✅ Good
FETCHReads full docs after an index matchOK; avoidable with covered queries
SORTIn-memory sort (no index for it)⚠️ Costly — add a sort index
  • winningPlan — the plan MongoDB chose to run.
  • totalDocsExamined vs nReturned — want these close; a big gap means a bad/absent index.
  • totalKeysExamined — index entries scanned.
Bottleneck checklist: See COLLSCAN? Add an index. See a SORT stage? Add the sort field to the index (ESR). docsExamined ≫ nReturned? Your index isn't selective enough.

25 TTL Indexes (Auto-Expiry)

A TTL (Time-To-Live) index automatically deletes documents after a set age — no cron job, and it cuts storage cost.

Create a TTL index
// Delete OTP docs 5 minutes after createdAt
db.otps.createIndex({ createdAt: 1 }, { expireAfterSeconds: 300 })

// Or expire at an exact per-doc time: set expireAfterSeconds: 0
// and store an explicit expiresAt Date on each document
db.sessions.createIndex({ expiresAt: 1 }, { expireAfterSeconds: 0 })
Use caseExpire after
OTP / 2FA codes5 minutes
User sessions1–24 hours
Password-reset tokens15–60 minutes

27 GridFS (Large Files)

A document is limited to 16 MB. GridFS stores larger files (videos, PDFs, high-res images) by splitting them into chunks across two collections: fs.files (metadata) and fs.chunks (data).

  const p = await Product.create(req.body);

});

  const filter = category ? { category } : {};
  const items = await Product.find(filter)
    .sort({ _id: -1 })

    .lean();               // plain objects = faster reads

  res.json(p);

  await Product.findByIdAndDelete(req.params.id);
  res.json({ deleted: true });
});

Document middleware
// Build a slug automatically before every save

// Query middleware: hide soft-deleted docs from every find

Problem 1 — Low stock alert CRUD

Task: Find all products with stock below 5 and mark them with status: "reorder" in a single command.

Show solution
Solution
db.products.updateMany(
  { stock: { $lt: 5 } },
  { $set: { status: "reorder" } }
)

Why: updateMany touches every matching doc; $lt filters by threshold; $set adds/overwrites the field.

Problem 2 — Second page of results Read

Task: Return page 2 of products (10 per page), sorted by price ascending.

Show solution
Solution
db.products.find()
  .sort({ price: 1 })
  .skip(10)   // (page - 1) * pageSize
  .limit(10)

Note: For very large collections, use range-based paging (price > lastSeenPrice) to avoid slow skip.

Problem 3 — Revenue by category Aggregation

Task: Compute total revenue (price × quantity) per category and show the top 3 categories.

Show solution
Solution
db.orders.aggregate([
  { $group: {
      _id: "$category",
      revenue: { $sum: { $multiply: ["$price", "$quantity"] } }
  }},
  { $sort: { revenue: -1 } },
  { $limit: 3 }
])

Why: $group aggregates; $multiply inside $sum computes a per-doc value before summing; $sort + $limit pick the top 3.

Problem 4 — Attach customer to each order $lookup

Task: For each order, attach the matching customer document from the customers collection.

Show solution
Solution
db.orders.aggregate([
  { $lookup: {
      from: "customers",
      localField: "customerId",
      foreignField: "_id",
      as: "customer"
  }},
  { $unwind: "$customer" }   // turn 1-element array into object
])

Why: $lookup joins; $unwind flattens the resulting single-element array for cleaner output.

Problem 5 — Prevent duplicate emails Index

Task: Guarantee no two users can share the same email at the database level.

Show solution
Solution
db.users.createIndex({ email: 1 }, { unique: true })

Why: A unique index rejects any insert/update that would duplicate the value — enforcement lives in the DB, not just app code.

Problem 6 — Add a tag without duplicates Arrays

Task: Add "featured" to a product's tags array only if it isn't already there.

Show solution
Solution
db.products.updateOne(
  { _id: 1 },
  { $addToSet: { tags: "featured" } }
)

Why: $addToSet is idempotent — running it twice won't create duplicates, unlike $push.

23 Top 50+ Interview Questions

60+ of the most frequently asked MongoDB questions with concise, interview-ready answers. Click any question to reveal the answer.

What is MongoDB and how does it differ from a relational database?

MongoDB is a document-oriented NoSQL database that stores data as flexible BSON documents in collections, rather than rows in fixed-schema tables. It favors embedding related data over JOINs, supports dynamic schemas, and scales horizontally via sharding.

What is BSON and why does MongoDB use it instead of JSON?

BSON (Binary JSON) is a binary-encoded superset of JSON. MongoDB uses it because it's faster to parse/traverse, more space-efficient, and supports extra types like Date, ObjectId, Decimal128, and Binary that plain JSON lacks.

What is the _id field?

Every document has a unique primary key called _id. If you don't supply one, MongoDB auto-generates a 12-byte ObjectId (timestamp + machine + process + counter). It's automatically indexed.

Embedding vs referencing — when do you use each?

Embed when data is accessed together and bounded in size (one-to-few), e.g. a post and its comments — one read gets everything. Reference when data is large, grows unbounded, or is shared across many parents (one-to-many / many-to-many), e.g. orders pointing to a customer _id.

What is an aggregation pipeline?

A framework for multi-stage data processing. Documents flow through ordered stages ($match, $group, $sort, $project, $lookup, etc.), each transforming the previous stage's output — used for grouping, computed fields, and joins.

Difference between find() and aggregate()?

find() retrieves and filters documents. aggregate() can additionally transform, group, compute, join ($lookup), and reshape data across multiple stages — far more powerful for analytics.

How do indexes work and what's the downside?

Indexes store an ordered structure (B-tree) of field values so queries can skip full collection scans. The trade-off: they consume RAM/disk and slow down writes because each write must also update the indexes. Index only fields you query or sort on.

What's a compound index and the ESR rule?

A compound index covers multiple fields in a defined order. The ESR rule says to order fields as Equality → Sort → Range for optimal performance, because MongoDB can use the index prefix efficiently for equality matches and sorting.

Does MongoDB support ACID transactions?

Yes. Single-document operations have always been atomic. Since v4.0, MongoDB supports multi-document ACID transactions (across collections/shards in v4.2+) via sessions — useful for operations like transfers that must all succeed or all fail.

Explain replication vs sharding.

Replication (replica set) keeps copies of data on multiple nodes for high availability and automatic failover. Sharding partitions data across servers by a shard key for horizontal scaling of storage and throughput. They're often combined.

What is an upsert?

An update with { upsert: true }: if a matching document exists it's updated; if not, a new document is inserted. Handy for "insert or update" logic in one call.

Difference between $push and $addToSet?

Both add to an array. $push always appends (allows duplicates); $addToSet only adds the value if it's not already present (keeps the array a set).

What is the document size limit?

A single BSON document is limited to 16 MB. For larger binary data, use GridFS, which splits files into chunks across two collections.

How do you improve a slow query?

Run .explain("executionStats") to inspect the plan. Look for COLLSCAN (full scan) and add an appropriate index; use projections to return only needed fields; follow the ESR rule for compound indexes; and avoid large skip offsets.

What is $elemMatch used for?

It matches array documents where a single element satisfies multiple conditions at once — e.g. an order item that is both qty > 1 and price < 50 in the same element.

What is a capped collection?

A fixed-size collection that keeps insertion order and automatically overwrites the oldest documents when full — great for logs or a rolling buffer. Created with db.createCollection("logs", { capped: true, size: 100000 }).

What is a TTL index?

A special single-field index on a Date field with expireAfterSeconds. MongoDB automatically deletes documents once they exceed that age — perfect for sessions, OTPs, and temporary caches, and it reduces storage cost.

What is a multikey index?

An index on an array field. MongoDB creates an index entry for each array element, so a query like { tags: "sale" } can use it. You can't create a compound index on two array fields.

What is a covered query?

A query where all requested fields exist in the index, so MongoDB answers it entirely from the index without reading documents. Requires the projection to exclude _id unless _id is in the index. Fastest possible read.

What is the difference between updateOne, replaceOne, and findOneAndUpdate?

updateOne applies operators like $set to specific fields. replaceOne swaps the entire document (except _id). findOneAndUpdate updates and returns the document (before or after, via returnDocument).

Difference between count(), countDocuments() and estimatedDocumentCount()?

countDocuments(filter) returns an accurate count using the filter. estimatedDocumentCount() reads collection metadata — very fast but approximate and ignores filters. Legacy count() is deprecated.

What is the WiredTiger storage engine?

MongoDB's default storage engine since 3.2. It provides document-level concurrency (locking), compression (snappy/zstd), and a cache for the working set — key to both performance and storage cost.

What is the "working set"?

The portion of data + indexes actively accessed by your queries. For good performance the working set should fit in RAM; if it doesn't, MongoDB hits disk frequently (page faults) and slows down.

What is a write concern?

It controls the acknowledgment level for writes: w:1 (primary ack), w:"majority" (majority of replica set), and j:true (journal on disk). Higher durability trades off latency.

What is a read concern / read preference?

Read concern controls consistency (e.g. "majority" returns acknowledged data). Read preference chooses which node to read from (primary, secondaryPreferred, etc.) — used to offload analytics to secondaries.

What is a shard key and what makes a good one?

The field(s) used to partition data across shards. A good shard key has high cardinality, low frequency (no hot value), and monotonic-free growth so writes spread evenly. A poor key creates a "hot shard".

Ranged vs hashed sharding?

Ranged keeps nearby key values together (good for range queries, risky for monotonic keys). Hashed distributes writes evenly by hashing the key (good for even load, bad for range queries).

What is $unwind?

An aggregation stage that deconstructs an array field, outputting one document per element. Commonly used before $group to aggregate over array items, or after $lookup to flatten the joined array.

Difference between $project and $addFields?

$addFields adds/updates fields while keeping all existing ones. $project reshapes the document, including only the fields you specify (plus any computed ones) — it can both include and exclude.

What is $facet?

An aggregation stage that runs multiple sub-pipelines on the same input in parallel, returning all results in one document — perfect for a dashboard that needs counts, top-N, and a histogram in a single query.

What is $merge vs $out?

Both write pipeline results to a collection. $out replaces the target collection entirely. $merge can insert, merge, or replace matching docs into an existing collection — ideal for incremental materialized views.

What is GridFS?

A specification for storing files larger than the 16 MB document limit. It splits a file into chunks stored in fs.chunks with metadata in fs.files, letting you stream large files in and out.

What is optimistic concurrency / how to avoid race conditions?

Use atomic operators ($inc, $set) instead of read-modify-write, add a version field checked in the filter, or use findOneAndUpdate — so two concurrent updates don't clobber each other.

What is the aggregation $group accumulator list?

Common accumulators: $sum, $avg, $min, $max, $first, $last, $push (collect into array), and $addToSet (unique array).

How do you model a many-to-many relationship?

Store an array of referenced _ids on one or both sides (e.g. a student with a courseIds array), or use a dedicated join collection for very large/attributed relationships. Resolve with $lookup.

What is the outlier / bucket pattern?

Design patterns for unbounded arrays. Bucketing groups many items into fixed-size documents (e.g. one doc per hour of sensor data). The outlier pattern handles rare huge documents separately so the common case stays small.

What is the difference between $in and $or?

$in matches one field against a list of values (concise, index-friendly). $or combines independent conditions across different fields. Prefer $in when checking the same field.

How does MongoDB ensure high availability?

Via replica sets: if the primary goes down, remaining members hold an election and promote a secondary within seconds. Drivers automatically reconnect to the new primary.

What is an ObjectId made of?

12 bytes: a 4-byte timestamp, 5-byte random value (machine/process), and a 3-byte incrementing counter. It's roughly sortable by creation time, so sorting by _id ≈ sorting by insert order.

How do you paginate efficiently on large collections?

Avoid large skip() — it scans skipped docs. Use range/keyset pagination: find({ _id: { $gt: lastId } }).sort({ _id: 1 }).limit(n), remembering the last _id from the previous page.

What is explain() and its verbosity modes?

It returns the query plan. Modes: queryPlanner (chosen plan), executionStats (actual timing/docs examined), and allPlansExecution (all considered plans). Use it to confirm an index is used.

What is a partial index and why use it?

An index built only on documents matching a filter (partialFilterExpression). It's smaller and cheaper — e.g. index only status: "active" users instead of the whole collection.

What is index cardinality and why does it matter?

Cardinality is the number of distinct values in a field. High-cardinality fields (like email) make selective, efficient indexes; low-cardinality fields (like boolean) filter poorly and may not benefit from an index.

What is the difference between a database and a collection?

A database is a container of collections (like a SQL schema). A collection is a group of documents (like a table). Both are created lazily on first write.

How do you do a case-insensitive search?

Use a regex with the i flag ({ name: /laptop/i }), or create a collation-enabled index with strength 2 for efficient case-insensitive matching, or use a text index.

What is a text index vs Atlas Search?

A text index supports basic keyword search with stemming. Atlas Search (Lucene-based) adds fuzzy matching, autocomplete, relevance scoring, and faceting — far more powerful for real search features.

What is a change stream?

A real-time feed of data changes (insert/update/delete) using db.collection.watch(). Apps subscribe to react instantly — e.g. live dashboards, cache invalidation, or syncing to another system.

What is connection pooling and why does it matter?

The driver keeps a pool of reusable connections. You should create one MongoClient at app startup and reuse it — creating a client per request exhausts the pool and kills throughput (a common serverless bug).

How do you handle schema migrations in MongoDB?

Because schemas are flexible, use incremental strategies: add new fields with defaults, run a background updateMany to backfill, and handle both old/new shapes in code (a schemaVersion field) during the transition.

What are common MongoDB anti-patterns?

Unbounded arrays, massive documents near 16 MB, storing one document per key like a SQL row, over-indexing, using $where/JS, huge skip() pagination, and building a separate collection per user/tenant.

What is the difference between deleteOne and findOneAndDelete?

deleteOne removes a document and returns only a count. findOneAndDelete removes it and returns the deleted document — useful when you need its contents afterward.

How do you secure a MongoDB deployment?

Enable authentication & role-based access control, use TLS, bind to private networks / IP allow-lists, apply least-privilege users, keep credentials in secrets/env vars, and disable server-side JavaScript if unused.

What is the aggregation pipeline memory limit?

Each stage is limited to 100 MB of RAM by default. For larger sorts/groups, enable { allowDiskUse: true } so stages can spill to disk.

What is the difference between horizontal and vertical scaling here?

Vertical = bigger server (more RAM/CPU) — simple but capped. Horizontal = sharding across many servers — MongoDB's strength for scaling beyond one machine's limits.

When would you NOT use MongoDB?

When you need complex multi-table transactions with heavy relational integrity, rigid schemas with lots of ad-hoc JOINs, or strong normalized reporting — a relational database may fit better. MongoDB shines with flexible, hierarchical, high-volume data.

How do you reduce MongoDB cost in production?

Use TTL indexes to auto-purge stale data, archive cold data to a cheaper tier (Atlas Online Archive), drop unused indexes ($indexStats), project only needed fields to cut transfer, right-size the cluster to the working set, and rely on built-in compression.

What is idempotency and which operators are idempotent?

An idempotent operation gives the same result if applied multiple times. $set and $addToSet are idempotent; $inc and $push are not (they change the result each run). Important for retries.

How do you back up and restore MongoDB?

Use mongodump/mongorestore for logical backups, filesystem/volume snapshots for large clusters, or Atlas continuous/cloud backups with point-in-time restore.

What is the difference between save() and insertOne() in Mongoose?

doc.save() works on a model instance and runs validation + save middleware; it inserts or updates depending on whether the doc is new. Model.create()/insertMany() are shortcuts that build and save in one call.

How do transactions work with Mongoose middleware?

Start a session with mongoose.startSession(), wrap operations in session.withTransaction(), and pass { session } to each query. Middleware still runs, and all writes commit or roll back together.

What is lean() in Mongoose and why use it?

.lean() returns plain JS objects instead of full Mongoose documents — skipping hydration, getters, and change tracking. It's significantly faster and lighter for read-only queries.

How do you enforce data integrity without JOINs?

Embed related data so it's atomic within one document, use schema validation, unique indexes for constraints, and application-level or transaction-based checks for cross-collection consistency.

What is the difference between $set and $setOnInsert?

In an upsert, $set always applies. $setOnInsert applies its fields only when a new document is inserted, not on updates — perfect for setting createdAt once.

24 Scenario-Based Questions

Design-and-debug style questions like those asked in senior interviews. Think through each, then expand the approach.

Scenario 1 — Instagram-style feed Modeling

Q: Design collections for users, posts, and comments where feeds load fast. Would you embed or reference comments?

Show approach

Reference posts from users (a post stores userId). For comments, embed the latest few in the post for instant feed rendering, and store the full list in a separate comments collection referenced by postId. This keeps feed reads to one query while comments stay unbounded and paginated on demand.

Post document
{
  _id: ObjectId,
  userId: ObjectId,
  text: "...",
  likeCount: 120,
  recentComments: [ { user, text } ], // last 3, embedded
  createdAt: ISODate
}
Scenario 2 — Sudden slow API Bottleneck

Q: A previously fast endpoint is now slow after data growth. How do you find and fix it?

Show approach

Run the query with .explain("executionStats"). A COLLSCAN with totalDocsExamined ≫ nReturned signals a missing index. Add a compound index following ESR, add projections to trim payload, and re-check the plan shows IXSCAN. Also verify the working set still fits RAM.

Scenario 3 — Money transfer Transactions

Q: Deduct from account A and credit account B. How do you guarantee both happen or neither?

Show approach

Wrap both updateOne calls in a multi-document transaction using a session. Use $inc for atomic balance changes and add a filter guard (balance: { $gte: amount }) so you never go negative. Commit on success, abort on any error.

Scenario 4 — IoT sensor data Time-series

Q: Millions of sensor readings per hour. How do you store them affordably and query recent data fast?

Show approach

Use a time-series collection (or the bucket pattern: one doc per sensor per hour holding an array of readings). Add a TTL index to auto-expire old data, archive cold data to a cheaper tier, and index on { sensorId: 1, timestamp: -1 } for recent-first queries.

Scenario 5 — Prevent double booking Concurrency

Q: Two users try to book the last seat simultaneously. How do you prevent overselling?

Show approach

Use a single atomic findOneAndUpdate with a filter that only matches if the seat is still free ({ seatId, booked: false }) and sets booked: true. Only one operation will match; the other returns null and is rejected — no race condition.

Scenario 6 — Multi-tenant SaaS Design

Q: How do you isolate data for thousands of tenants without a collection per tenant?

Show approach

Add a tenantId field to every document and make it the first field of every index (and often the shard key). All queries filter by tenantId. This scales far better than per-tenant collections/databases, which cause metadata bloat.

Scenario 7 — Leaderboard / top-N Performance

Q: A game needs the top 100 players by score, refreshed constantly for millions of users. How?

Show approach

Index { score: -1 } and serve find().sort({score:-1}).limit(100) — a covered/indexed query. For extreme scale, maintain a pre-computed leaderboard document updated with $inc, or use a cache (Redis) fed by change streams.

Scenario 8 — Nearby restaurants Geospatial

Q: Find all restaurants within 2 km of a user's location, nearest first.

Show approach
2dsphere + $near
db.restaurants.createIndex({ location: "2dsphere" })
db.restaurants.find({
  location: { $near: {
    $geometry: { type: "Point", coordinates: [lng, lat] },
    $maxDistance: 2000   // metres
  }}
})
Scenario 9 — Idempotent payment webhook Reliability

Q: A payment provider may send the same webhook twice. How do you avoid double-processing?

Show approach

Store each event's provider eventId with a unique index. Insert it first; a duplicate insert throws a duplicate-key error, so you skip re-processing. Or use updateOne(..., { upsert: true }) with $setOnInsert to process exactly once.

Scenario 10 — Growing analytics collection Cost

Q: An events collection grows by 50M docs/month and the bill is climbing. How do you control it?

Show approach

Add a TTL index to auto-delete raw events after N days, roll up daily aggregates with $merge into a small summary collection, move cold data to Atlas Online Archive, drop unused indexes, and right-size the cluster to the working set.
