Overview

REST (Representational State Transfer) is an architectural style for networked APIs over HTTP. Resources are identified by URLs and manipulated with standard HTTP methods.

  • Resource-oriented — everything is a resource with a URL.
  • Uses HTTP semantics — methods, status codes, headers.
  • Stateless — each request carries all needed context.
  • Representation-based — usually JSON.
Interview importance: HTTP methods/status codes, idempotency, versioning, auth, and caching are the core REST interview topics.

Richardson Maturity Model: L0 (single endpoint) → L1 (resources) → L2 (HTTP verbs + status) → L3 (HATEOAS/hypermedia). Most "REST" APIs are Level 2.

Resources & URI Design

GoodAvoid

GET /users/42/ordersGET /users?getOrders=42
POST /usersPOST /createUser
  • Nouns, not verbs — the method is the verb.
  • Plural collections — /users, /users/{id}.
  • Nest for relationships — /users/{id}/orders (avoid deep nesting > 2 levels).
  • Lowercase, hyphenated — /order-items.

Idempotency & Safety

Safe = no side effects (GET). Idempotent = repeating gives the same result (GET, PUT, DELETE). Critical for reliable retries.

Idempotency-Key for POST
POST /payments
Idempotency-Key: 4f8c-...-a1

# Server stores the key + result. A retry with the same key
# returns the original response instead of charging twice.
Why it matters: Networks retry. Idempotent design (or idempotency keys) prevents duplicate orders/charges.

Versioning

StrategyExample
URI path/api/v1/users (most common)
HeaderAccept: application/vnd.api.v2+json
Query param/users?version=2
Rule: Never break existing clients. Add fields (non-breaking) freely; version only for breaking changes, and deprecate old versions with a timeline + Deprecation/Sunset headers.

Pagination, Filtering, Sorting

Query conventions
GET /products?category=phones&price_gte=100    # filter
GET /products?sort=-price,name                 # sort (desc price, asc name)
GET /products?page=2&limit=20                   # offset pagination
GET /products?cursor=eyJpZCI6MTIzfQ&limit=20    # cursor pagination

Offset

  • Simple page/limit
  • Slow on deep pages; can skip/duplicate on writes

Cursor (keyset)

  • Stable & fast for large/real-time data
  • Opaque cursor from the last item

Error Handling (RFC 7807)

Problem Details format
HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json
{
  "type": "https://api.com/errors/validation",
  "title": "Validation failed",
  "status": 422,
  "detail": "email is invalid",
  "errors": [{ "field": "email", "message": "invalid" }]
}
Consistency: Use one error envelope everywhere with a machine-readable type/code, a human message, and field-level details. Never leak stack traces.

Rate Limiting

  • Return 429 Too Many Requests with Retry-After.

Best Practices

  • Nouns + plural collections; correct HTTP methods & status codes.
  • Version the API; never break existing clients.
  • Consistent error format (RFC 7807); validate all input.
  • Paginate list endpoints; support filtering/sorting.
  • Idempotency keys for unsafe retries.
  • Cache with ETag/Cache-Control; rate-limit.
  • Secure with HTTPS, auth, scopes, OWASP mitigations.
  • Document with OpenAPI; return Location on create.

HTTP Internals

Request lifecycle

DNS resolve → TCP handshake → (TLS handshake) → send request → server processes → response → connection reused or closed.

FeatureWhat it does
Keep-AliveReuse one TCP connection for many requests (HTTP/1.1 default) — avoids handshake cost
HTTP/2 multiplexingMany concurrent streams over one connection; header compression (HPACK); fixes head-of-line blocking at the app layer
HTTP/3 (QUIC)Runs over UDP; removes TCP head-of-line blocking; faster connection setup (0-RTT)
Chunked transferTransfer-Encoding: chunked streams a response without knowing its length up front
Interview point: HTTP/2 multiplexing means you no longer need domain sharding/sprite hacks; one connection carries many parallel requests efficiently.

Conditional Requests & Optimistic Concurrency

Conditional headers prevent lost updates and save bandwidth by acting only if a precondition holds.

Prevent lost updates with If-Match (ETag)
GET /docs/9          -> 200  ETag: "v3"

PUT /docs/9
If-Match: "v3"       # only update if still version 3
<- 200 OK            # success, new ETag "v4"
<- 412 Precondition Failed  # someone else changed it first
HeaderUse
If-None-MatchGET returns 304 if unchanged (caching)
If-MatchWrite only if ETag matches (concurrency)
If-Unmodified-SinceWrite only if not changed since a date
Lost update problem: Two clients read v3, both write — the second silently overwrites the first. If-Match makes the stale write fail with 412 instead.

API Resilience

PatternPurpose
TimeoutsFail fast instead of hanging on a slow dependency
Retries + exponential backoff + jitterRecover from transient failures without thundering herd
Circuit breakerStop calling a failing service; fail fast; auto-recover
Idempotent retriesSafe to retry writes (idempotency keys)
Request deduplicationDrop duplicate in-flight/replayed requests
BulkheadIsolate resource pools so one dependency can't sink all
Client retry with backoff + jitter
for (attempt = 0; attempt < 3; attempt++) {
  res = await call();
  if (res.ok || res.status < 500) break;   // don't retry 4xx
  await sleep(rand() * 2 ** attempt * 100); // backoff + jitter
}
Honor Retry-After on 429/503 and only retry idempotent/safe operations (or those with an idempotency key).

Webhooks & Async APIs

Webhooks (push)

  • Server calls your URL when an event happens
  • Real-time, no polling overhead
  • Needs a public endpoint + retries

Polling (pull)

  • Client repeatedly asks "any updates?"
  • Simple; wasteful; delayed
  • Use with cursors/ETags
Verify a webhook signature (HMAC)
// Provider signs the payload; you verify to prevent spoofing
const expected = hmacSHA256(secret, rawBody);
if (!timingSafeEqual(expected, req.headers["x-signature"]))
  return res.sendStatus(401);
// respond 2xx fast; process async; dedupe by event id (idempotent)
Async long-running job (202 + polling)
POST /reports            -> 202 Accepted  Location: /reports/7/status
GET  /reports/7/status   -> 200 { state: "processing" }
GET  /reports/7/status   -> 303 See Other  Location: /reports/7
Webhook best practices: verify signatures, respond 2xx quickly (process async), retry with backoff, and make handlers idempotent (dedupe by event ID).

Design Scenarios

Scenario 1 — Design a payments endpoint Idempotency

Q: Clients may retry on timeout. How do you avoid double charges?

Show approach

Require an Idempotency-Key header; store key → result. On retry with the same key, return the original response. Use 201 + Location; 409 on conflicting keys.

Scenario 2 — Users can read others' data Security

Q: GET /orders/{id} returns any order regardless of owner.

Show approach

Broken Object-Level Authorization (BOLA). Always verify the authenticated user owns/*may access* the resource, not just that they're logged in.

Scenario 3 — A list endpoint times out Performance

Q: GET /events returns millions of rows.

Show approach

Enforce pagination (cursor-based for large data), add filters, indexes, field selection, and caching. Never return unbounded lists.

Top Interview Questions

What is REST?

An architectural style using HTTP where resources have URLs and are manipulated with standard methods; stateless, cacheable, uniform interface.

PUT vs PATCH vs POST?

PUT replaces the full resource (idempotent); PATCH partially updates; POST creates or performs non-idempotent actions.

What does idempotent mean?

Repeating the request has the same effect as doing it once — GET, PUT, DELETE. Enables safe retries; POST needs an idempotency key.

Safe vs idempotent methods?

Safe = no side effects (GET, HEAD). Idempotent = repeatable without extra effect (GET, PUT, DELETE). POST is neither.

201 vs 200 vs 204?

201 Created (with Location) for successful creation; 200 OK for general success with a body; 204 No Content for success without a body (e.g. DELETE).

401 vs 403?

401 = not authenticated; 403 = authenticated but not authorized.

How do you version an API?

URI path (/v1), header, or query param. Add non-breaking changes freely; version for breaking changes and deprecate with a timeline.

What is HATEOAS?

Hypermedia As The Engine Of Application State — responses include links to related actions, so clients navigate the API dynamically (REST maturity Level 3).

Offset vs cursor pagination?

Offset (page/limit) is simple but slow/unstable on large or changing data. Cursor (keyset) uses the last item's key — stable and fast.

How do you cache REST responses?

Cache-Control for freshness and ETag + If-None-Match for revalidation (304). CDNs cache public responses.

How do you secure a REST API?

HTTPS, token auth (JWT/OAuth2) with scopes, object-level authorization (avoid BOLA), input validation, rate limiting, CORS allow-list, OWASP mitigations.

REST vs GraphQL?

REST has fixed endpoints and native HTTP caching; GraphQL lets clients request exactly the fields they need (no over/under-fetching) but caching is harder.

What is content negotiation?

The client's Accept header tells the server which representation/version/format to return for the same resource.

Why is statelessness important?

Each request is self-contained, so any server instance can handle it — enabling horizontal scaling and simpler load balancing.

How do you handle partial failures / retries?

Idempotent design + idempotency keys, exponential backoff on the client, and clear 4xx/5xx codes with Retry-After where relevant.

HTTP/1.1 vs HTTP/2 vs HTTP/3?

1.1 added Keep-Alive; HTTP/2 multiplexes many streams over one connection with header compression; HTTP/3 runs over QUIC/UDP, removing TCP head-of-line blocking and speeding setup.

What is Keep-Alive?

Reusing a single TCP connection for multiple requests to avoid repeated handshakes — default in HTTP/1.1.

What is chunked transfer encoding?

Transfer-Encoding: chunked streams a response in pieces without a known total length — used for streaming/generated content.

How do you prevent lost updates?

Optimistic concurrency with ETags + If-Match: a stale write fails with 412 Precondition Failed instead of silently overwriting.

If-None-Match vs If-Match?

If-None-Match is for caching (304 if unchanged); If-Match is for concurrency (write only if the ETag still matches).

What is a circuit breaker?

A resilience pattern that stops calling a failing dependency after a threshold (fail fast / fallback), then probes to recover — preventing cascading failures.

Why add jitter to backoff?

Randomized delays prevent many clients from retrying in lockstep (thundering herd) and overwhelming a recovering service.

Webhooks vs polling?

Webhooks push events to your URL in real time (efficient, needs a public endpoint + signature verification); polling repeatedly asks for updates (simple but wasteful/delayed).

How do you secure webhooks?

Verify an HMAC signature over the raw body with a shared secret (timing-safe compare), respond 2xx fast, process async, and dedupe by event ID (idempotent).

How do you design a long-running operation?

Return 202 Accepted with a status URL; the client polls the status endpoint, which returns progress and finally 303/200 with the result.

How do you dedupe duplicate requests?

Use an idempotency key or event ID; store processed keys and short-circuit repeats, returning the original result.

Which errors are safe to retry?

Transient 5xx (500/502/503/504) and 429 (respect Retry-After). Don't retry 4xx like 400/401/403/404/422 — the request itself is wrong.
