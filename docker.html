Overview

Docker packages an app and all its dependencies into a portable image that runs identically anywhere as a container — solving "works on my machine".

🐳 Containers

  • Share the host kernel (lightweight)
  • Start in milliseconds
  • MBs in size
  • Process-level isolation

🖥️ Virtual Machines

  • Full guest OS (heavy)
  • Boot in seconds/minutes
  • GBs in size
  • Hardware-level isolation
Interview importance: image vs container, Dockerfile layers/caching, multi-stage builds, volumes vs bind mounts, and Compose are the most-asked Docker topics.

Images & Containers

TermMeaning
ImageRead-only template (layers) built from a Dockerfile
ContainerA running instance of an image (writable layer on top)
RegistryStores/distributes images (Docker Hub, ECR, GHCR)
LayerA filesystem diff cached & shared between images
VolumePersistent data managed by Docker
Flow: Dockerfile → docker build → image → docker run → container. Push/pull images via a registry.

Core CLI

Everyday commands
docker build -t myapp:1.0 .        # build image from Dockerfile
docker run -d -p 3000:3000 --name web myapp:1.0
docker ps        / docker ps -a  # running / all containers
docker logs -f web
docker exec -it web sh            # shell into a container
docker stop web  / docker rm web
docker images   / docker rmi myapp:1.0
docker system prune -af          # clean up (⚠️ removes unused)

Dockerfile

Anatomy
FROM node:20-alpine        # base image
WORKDIR /app               # working dir inside container
COPY package*.json ./       # copy deps manifest first (cache!)
RUN npm ci                  # install (own layer)
COPY . .                    # copy source
EXPOSE 3000                 # document port
ENV NODE_ENV=production
CMD ["node", "server.js"]     # default process
InstructionPurpose
FROMBase image
WORKDIRSet working directory
COPY / ADDAdd files (ADD can fetch URLs/untar)
RUNExecute at build time (new layer)
CMD vs ENTRYPOINTDefault command vs fixed executable
EXPOSEDocument a port (doesn't publish)
ENV / ARGRuntime env / build-time arg
CMD vs ENTRYPOINT: ENTRYPOINT sets the executable, CMD provides default args. Use both: ENTRYPOINT ["node"] + CMD ["server.js"].

Layers & Build Caching

Each instruction creates a cached layer. If a layer's inputs don't change, Docker reuses the cache — so order matters.

Cache-friendly ordering
# ✅ deps copied & installed BEFORE source
COPY package*.json ./
RUN npm ci            # cached unless deps change
COPY . .              # source changes don't bust the deps layer

# ❌ copying everything first re-installs deps on every code change
Use a .dockerignore (node_modules, .git, .env, dist) to keep the build context small and avoid leaking secrets.

Multi-Stage Builds

Build in one stage, ship only the artifacts in a tiny final image — drastically smaller and more secure (no build tools in production).

Generic multi-stage
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build            # produces /app/dist

FROM nginx:alpine           # small final image, no Node
COPY --from=build /app/dist /usr/share/nginx/html
Result: the final image contains only the built assets + runtime — often 10× smaller than a single-stage build.

Volumes & Storage

Named Volume

Docker-managed persistent storage (DB data). Survives container removal.

volume
docker run -v pgdata:/var/lib/postgresql/data postgres

Bind Mount

Maps a host path into the container (live code in dev).

bind
docker run -v $(pwd):/app node
Container filesystems are ephemeral — data written inside a container is lost on removal. Persist databases/uploads with named volumes.

Networking

DriverUse
bridgeDefault; containers on the same bridge talk by name
hostShare the host network (no isolation)
noneNo networking
overlayMulti-host (Swarm/K8s)
Service discovery by name
docker network create app-net
docker run --network app-net --name db postgres
docker run --network app-net --name api myapi
# api connects to "db:5432" — Docker DNS resolves the name
Port publishing: -p host:container exposes a container port to the host. EXPOSE alone does not publish.

Environment & Secrets

Env vars & files
docker run -e NODE_ENV=production -e PORT=3000 myapp
docker run --env-file .env myapp   # load from file
Never bake secrets into images (they persist in layers/history). Use runtime env vars, Docker/BuildKit secrets, or a secrets manager. Add .env to .dockerignore.

Docker Compose

Compose defines and runs multi-container apps from one YAML file — docker compose up starts everything.

compose.yaml
services:
  api:
    build: ./api
    ports: ["3000:3000"]
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/app
    depends_on: [db]
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
    volumes: [pgdata:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 5s
volumes:
  pgdata:
Compose commands
docker compose up -d --build
docker compose logs -f api
docker compose down -v          # stop + remove volumes

Real-World: Dockerize a React App

React builds to static files, so the pattern is: build with Node, serve with Nginx (multi-stage → tiny image).

react-app/Dockerfile
# --- Stage 1: build ---
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build              # Vite -> /app/dist (CRA -> /app/build)

# --- Stage 2: serve ---
FROM nginx:1.27-alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
nginx.conf (SPA fallback)
server {
  listen 80;
  location / {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /index.html;   # client-side routing
  }
}
Build & run
docker build -t react-app .
docker run -p 8080:80 react-app     # open http://localhost:8080
Dev mode (hot reload): use a single Node stage + bind mount and run npm run dev -- --host; use the Nginx multi-stage image for production.

Real-World: Dockerize a FastAPI App

fastapi-app/Dockerfile
FROM python:3.12-slim
WORKDIR /app
ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
RUN adduser --disabled-password app && chown -R app /app
USER app                     # non-root
EXPOSE 8000
# Gunicorn + Uvicorn workers for production
CMD ["gunicorn", "main:app", "-k", "uvicorn.workers.UvicornWorker", "-w", "4", "-b", "0.0.0.0:8000"]
Build & run
docker build -t fastapi-app .
docker run -p 8000:8000 --env-file .env fastapi-app
# docs at http://localhost:8000/docs
Slim + no-cache: use python:3.12-slim, --no-cache-dir, and a non-root user. For dev, mount code and run uvicorn main:app --reload --host 0.0.0.0.

Real-World: Full-Stack with Compose

React (Nginx) + Express/FastAPI API + Postgres + Redis, wired together and started with one command.

compose.yaml (full stack)
services:
  web:
    build: ./react-app          # Nginx-served React
    ports: ["8080:80"]
    depends_on: [api]
  api:
    build: ./express-app        # or ./fastapi-app
    ports: ["3000:3000"]
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/app
      - REDIS_URL=redis://cache:6379
    depends_on:
      db: { condition: service_healthy }
      cache: { condition: service_started }
  db:
    image: postgres:16
    environment: { POSTGRES_USER: user, POSTGRES_PASSWORD: pass, POSTGRES_DB: app }
    volumes: [pgdata:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 5s
  cache:
    image: redis:7-alpine
volumes:
  pgdata:

🌍 One command up

docker compose up -d --build builds the React & API images, starts Postgres + Redis with health gating, wires service discovery by name (db, cache), and serves the SPA on :8080 talking to the API on :3000 — a complete reproducible environment.

Registry & CI/CD

Tag, push, pull
docker tag myapp:1.0 ghcr.io/user/myapp:1.0
docker login ghcr.io
docker push ghcr.io/user/myapp:1.0
docker pull ghcr.io/user/myapp:1.0
CI/CD: build + scan + push on each commit (GitHub Actions), tag with the git SHA + latest, and deploy the immutable image. Use BuildKit caching to speed builds.

Orchestration (Kubernetes intro)

Compose is great for one host. For scaling across many nodes with self-healing, rolling updates, and service discovery, use an orchestrator.

ConceptMeaning
PodSmallest deployable unit (1+ containers)
DeploymentDeclarative replicas + rolling updates
ServiceStable network endpoint / load balancing
IngressHTTP routing into the cluster
ConfigMap/SecretConfig & secrets injection
Docker Swarm is the simpler built-in orchestrator; Kubernetes is the industry standard for large-scale, self-healing deployments.

Scenario-Based Questions

Scenario 1 — Image is huge (1.5GB) Optimization

Q: Your Node/React image is enormous and slow to deploy.

Show approach

Use a multi-stage build (build with Node, serve static with Nginx), an alpine/slim base, a .dockerignore (exclude node_modules/.git), install prod-only deps (npm ci --omit=dev), and combine RUN layers. Result: 10× smaller.

Scenario 2 — Data lost on restart Volumes

Q: Your database loses all data when the container restarts.

Show approach

Container filesystems are ephemeral. Mount a named volume at the DB's data path (-v pgdata:/var/lib/postgresql/data) so data persists across container removal.

Scenario 3 — Rebuild reinstalls deps every time Caching

Q: Every code change triggers a full npm install.

Show approach

Copy package*.json and run install before copying the source, so the deps layer is cached and only busts when dependencies change.

Scenario 4 — Containers can't talk to each other Networking

Q: The API can't reach the database container.

Show approach

Put them on the same user-defined network (Compose does this automatically) and connect by service name (db:5432), not localhost — each container has its own network namespace.

Top Interview Questions

Image vs container?

An image is a read-only template (layers); a container is a running instance of an image with a writable top layer.

Container vs VM?

Containers share the host kernel (lightweight, fast, MBs); VMs run a full guest OS (heavy, slower, GBs). Containers isolate at the process level.

What is a multi-stage build?

Using multiple FROM stages to build in one and copy only artifacts into a small final image — no build tools shipped to production.

How does layer caching work?

Each instruction is a cached layer; if its inputs are unchanged, Docker reuses it. Order instructions from least- to most-frequently-changing (deps before source).

CMD vs ENTRYPOINT?

ENTRYPOINT sets the executable; CMD gives default arguments (overridable at run). Combine them for a fixed command with defaults.

Volume vs bind mount?

A named volume is Docker-managed persistent storage (best for DB data); a bind mount maps a host path (best for live code in dev).

How do containers communicate?

On a user-defined network they resolve each other by service/container name via Docker's built-in DNS; publish ports to reach the host.

How do you reduce image size?

Multi-stage builds, alpine/slim/distroless bases, .dockerignore, prod-only deps, fewer/combined layers, and cleaning caches in the same RUN.

What is docker-compose for?

Defining and running multi-container apps from one YAML file (services, networks, volumes, healthchecks) with docker compose up.

Why run as non-root?

To limit damage if the container is compromised — a root process could more easily escalate. Use a dedicated USER.

How do you handle secrets?

Never bake into images (layers are inspectable). Use runtime env vars, --env-file, BuildKit/Docker secrets, or a secrets manager; keep .env in .dockerignore.

What is a healthcheck?

A command Docker runs to report container health; orchestrators/Compose use it to gate dependents and restart unhealthy containers.

EXPOSE vs -p?

EXPOSE only documents a port; -p host:container actually publishes it to the host.

What is .dockerignore?

Like .gitignore for the build context — excludes files (node_modules, .git, .env) to speed builds and avoid leaking secrets into layers.

How do you dockerize a React app?

Multi-stage: build with Node (npm run build), then serve the static output with Nginx (with an SPA try_files fallback) in a tiny final image.
