Overview

Web accessibility (a11y) means building sites everyone can use — including people with visual, motor, auditory, or cognitive disabilities, and those using assistive technology.

  • Who benefits: screen reader users, keyboard-only users, low-vision, color-blind, motor-impaired, and situationally-limited users.
  • Why: inclusion, legal compliance (ADA, Section 508, EAA), SEO overlap, and better UX for all.
Interview importance: Semantic HTML, ARIA (and when NOT to use it), keyboard/focus management, and WCAG basics are the most-asked a11y topics.

WCAG & POUR Principles

WCAG (Web Content Accessibility Guidelines) is organized around four principles — POUR — with conformance levels A, AA (the common legal target), and AAA.

PrincipleMeansExamples
PerceivableContent can be perceivedAlt text, captions, contrast
OperableUI can be operatedKeyboard access, no traps, enough time
UnderstandableContent & UI are clearLabels, consistent nav, error help
RobustWorks with assistive techValid HTML, correct ARIA
Target AA: Most laws/contracts require WCAG 2.1/2.2 Level AA — e.g. 4.5:1 text contrast, keyboard operability, visible focus, labels, and captions.

Semantic HTML (First!)

The #1 rule: use the right native element. Semantic HTML gives you accessibility (roles, states, keyboard) for free.

✅ Do

native
<button onclick="save()">Save</button>
<nav> <main> <header> <footer>

❌ Avoid

div soup
<div class="btn" onclick="save()">Save</div>
<!-- not focusable, no role, no Enter/Space -->
Landmarks: header, nav, main, aside, footer let screen-reader users jump between regions. Use one <h1> and a logical heading order.

ARIA (Roles, States, Properties)

ARIA adds semantics when native HTML can't. It has three types: roles (what it is), states (current condition), properties (relationships).

Custom toggle with ARIA
<button role="switch" aria-checked="false" aria-label="Dark mode"></button>

<!-- relationships -->
<input aria-describedby="hint" aria-invalid="true">
<p id="hint">Min 8 characters</p>
TypeExamples
Rolesdialog, tablist, alert, switch, navigation
Statesaria-expanded, aria-checked, aria-selected, aria-hidden, aria-disabled
Propertiesaria-label, aria-labelledby, aria-describedby, aria-controls
First rule of ARIA: Don't use ARIA if a native element works. Bad ARIA is worse than none. ARIA changes semantics only — it does not add behavior (you still wire keyboard handling yourself).

Keyboard Navigation

KeyExpected
Tab / Shift+TabMove between focusable elements
Enter / SpaceActivate buttons/links
Arrow keysMove within a widget (menu, tabs, radio)
EscClose dialog/menu
  • Every interactive element must be reachable & operable by keyboard.
  • tabindex="0" makes a custom element focusable; -1 makes it focusable only programmatically. Avoid positive tabindex.
  • No keyboard traps — users must be able to tab out (except modals, which trap intentionally then release).

Focus Management

Modal focus trap + restore
// On open: remember trigger, move focus into dialog
lastFocused = document.activeElement;
dialog.querySelector("[autofocus], button").focus();

// Trap Tab within the dialog; Esc closes
// On close: restore focus to the trigger
lastFocused.focus();
  • Visible focus: never remove outlines without a replacement (:focus-visible).
  • Route changes (SPA): move focus to the new page's heading and announce it.
  • Skip link: a "Skip to content" link as the first focusable element.
Anti-pattern: outline: none with no alternative — keyboard users lose track of where they are.

Screen Readers

SRPlatform
NVDA / JAWSWindows
VoiceOvermacOS / iOS
TalkBackAndroid
  • They build an accessibility tree from the DOM + ARIA and announce roles/names/states.
  • Accessible name comes from: aria-labelledby → aria-label → associated <label> → text content → title.
  • aria-hidden="true" hides decorative content from SRs; .sr-only CSS visually hides but keeps it for SRs.
Visually-hidden but SR-readable
.sr-only {
  position: absolute; width: 1px; height: 1px;
  padding: 0; margin: -1px; overflow: hidden;
  clip: rect(0,0,0,0); white-space: nowrap; border: 0;
}

Live Regions

Live regions announce dynamic changes (form errors, toasts, cart updates) without moving focus.

aria-live
<div aria-live="polite">Saved!</div>       <!-- waits for a pause -->
<div aria-live="assertive" role="alert">Error!</div> <!-- interrupts -->
Use polite for status updates, assertive only for urgent messages. Keep the region in the DOM; change its text to trigger announcements.

Accessible Forms

Labels, errors, grouping
<label for="email">Email</label>
<input id="email" type="email" aria-describedby="err" aria-invalid="true" required>
<p id="err" role="alert">Enter a valid email</p>

<fieldset><legend>Shipping</legend> ... </fieldset>
  • Every input has a programmatic label (for/id, not placeholder-only).
  • Group related fields with fieldset/legend.
  • Associate errors with aria-describedby + aria-invalid; move focus to the first error on submit.

Images & Media

  • Informative images: descriptive alt. Decorative: alt="" (empty) so SRs skip them.
  • Complex images (charts): short alt + longer description nearby.
  • Video/audio: captions (required AA), transcripts, and audio descriptions.
  • Don't convey meaning with color alone or images of text.

Color & Contrast

ContentMin ratio (AA)
Normal text4.5 : 1
Large text (≥18.66px bold / 24px)3 : 1
UI components & graphics3 : 1
Don't rely on color alone for state (e.g. error = red). Add an icon, text, or pattern so color-blind users get the message.

Motion, Zoom & Touch

Respect reduced motion
@media (prefers-reduced-motion: reduce) {
  * { animation: none !important; transition: none !important; }
}
  • Support 200% zoom / 320px reflow without loss of content.
  • Touch targets ≥ 24×24px (44px recommended) with spacing.
  • Avoid content that flashes more than 3×/second (seizure risk).

React Accessibility

a11y in React
// use htmlFor + useId for label association
const id = useId();
<label htmlFor={id}>Name</label><input id={id}/>

// focus management on route change / modal open
useEffect(() => { headingRef.current?.focus(); }, [route]);

// fragments to avoid extra divs; real buttons/links
  • Use jsx-a11y ESLint plugin to catch issues at build.
  • Prefer real <button>/<a> over clickable divs.
  • Manage focus on mount/unmount of modals, drawers, toasts, and route changes.
  • Portals for modals keep DOM order sensible; trap + restore focus.

Best Practices

  • Semantic HTML first; ARIA only when needed (and correctly).
  • Everything keyboard-operable; visible focus; logical order.
  • Labels for all controls; associate & announce errors.
  • Alt text; captions; don't rely on color alone.
  • AA contrast; respect reduced motion; support zoom.
  • Manage focus for modals & SPA route changes.
  • Test with keyboard + a screen reader, not just automation.

Scenario-Based Questions

Scenario 1 — Custom dropdown unusable by keyboard Keyboard

Q: A div-based dropdown can't be opened or navigated without a mouse.

Show approach

Use a real <button> trigger with aria-expanded; make the listbox role="listbox" with options; handle Arrow/Enter/Esc keys; manage focus and aria-activedescendant. Better: use a native <select> or a vetted library.

Scenario 2 — Modal doesn't trap focus Focus

Q: Tabbing in a modal moves focus to the page behind it.

Show approach

Trap Tab within the dialog, set role="dialog" + aria-modal="true", move focus in on open, restore to the trigger on close, and close on Esc. Hide background from SRs with aria-hidden/inert.

Scenario 3 — Form errors not announced SR

Q: Screen-reader users don't know a submission failed.

Show approach

Put errors in a role="alert"/aria-live region, associate each with its field via aria-describedby + aria-invalid, and move focus to the first error or an error summary.

Top Interview Questions

What are the WCAG POUR principles?

Perceivable, Operable, Understandable, Robust — the four foundations, with conformance levels A, AA (typical target), AAA.

When should you use ARIA?

Only when native HTML can't express the semantics. First rule of ARIA: don't use ARIA if a native element works. Bad ARIA is worse than none.

Why is semantic HTML important?

Native elements provide roles, states, and keyboard behavior for free, and are announced correctly by screen readers.

How does an accessible name get computed?

Order: aria-labelledby → aria-label → associated label → text content → title.

tabindex values?

0 = focusable in normal order; -1 = focusable only via script; positive = avoid (breaks natural order).

How do you make a modal accessible?

role="dialog" + aria-modal, trap focus, move focus in on open, restore on close, Esc to close, hide the background from SRs.

polite vs assertive live regions?

Polite waits for a pause (status); assertive interrupts immediately (urgent errors). Keep the region in the DOM and update its text.

What's the minimum contrast ratio?

AA: 4.5:1 for normal text, 3:1 for large text and UI components/graphics.

How do you hide content correctly?

aria-hidden="true" hides from SRs (decorative); .sr-only hides visually but keeps it for SRs; display:none hides from both.

How do you handle focus in a SPA?

On route change, move focus to the new page's heading (and/or announce via a live region) so SR/keyboard users aren't stranded.

Placeholder vs label?

A placeholder is not a label — it disappears on input and isn't reliably announced. Always use a real associated <label>.

What is the accessibility tree?

A representation the browser builds from the DOM + ARIA that assistive tech consumes — exposing roles, names, states, and relationships.

How do you test accessibility?

Automated (axe/Lighthouse/jest-axe) for coverage, plus manual keyboard walkthrough and a screen reader — automation catches only ~30–40%.

How do you support reduced motion?

Honor prefers-reduced-motion to disable/limit animations for users prone to motion sickness/vestibular issues.

What is a skip link?

A "Skip to main content" link that's the first focusable element, letting keyboard users bypass repeated navigation.
